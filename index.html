SQL Quest: Architectural & Technical Guide

A comprehensive breakdown of the application architecture, database schemas, custom SQL parser logic, validation systems, and locking features.

🗺️ Architectural Overview

SQL Quest is built as a single-page application (SPA) with zero external server dependencies, running entirely client-side. The codebase consists of three main files:

  • index.html: Defines the 3-pane layout (sidebar roadmap/AI copilot, database browser, editor/results pane).
  • styles.css: Theme variables and visual aesthetics based on the Tech-Editorial Light design token system.
  • database.js: Custom in-browser parser/runner simulating table scans, where filtering, joins, and aggregates.
  • app.js: State machine controller (current quest, tab states, unlocks), editor syntax highlighter, and Real-Time Tutor keystroke parser.

💾 Space Academy Database Schema

The environment hosts four space-themed tables designed to teach basic SQL relational structures. All tables are defined inside database.js:

1. space_crew

Contains the roster of cadets, officers, and droids aboard the ship.

Column Type Description
idINTEGERUnique identifier (Primary Key)
nameVARCHARFull name of crew member
roleVARCHARProfessional role designation
statusVARCHARCurrent status (Active, Suspended, Retired)
years_activeINTEGERYears of service in the academy

2. planets

Holds astronomical details about discovered planets.

Column Type Description
idINTEGERPlanetary identifier (Primary Key)
nameVARCHARName of the planet
distance_lyFLOATDistance from home dock in Light-Years
typeVARCHARTerrestrial, Gas Giant, or Ice Giant composition
has_lifeBOOLEANTRUE if biosignatures detected

3. cargo

Tracks supplies stored inside the ship's freight compartments.

Column Type Description
idINTEGERCargo identifier (Primary Key)
itemVARCHARName of the cargo shipment item
categoryVARCHARClassification (Tech, Supplies, Fuel, Specimens)
weight_kgINTEGERMass of the items in kilograms
securedBOOLEANTRUE if locked into storage slots

4. space_missions

Tracks flight path routes, matching pilots back to the crew roster.

Column Type Description
idINTEGERMission identifier (Primary Key)
pilot_idINTEGERCrew pilot reference (Foreign Key matching space_crew.id)
destinationVARCHARTarget planetary destination
shipVARCHARHull name of spaceship assigned

⚙️ In-Browser Database Engine

The database simulator (database.js) utilizes strict regular expressions to process user queries locally:

  • Keyword Typos: Alerts users to spellings like SELEC or FORM before failure.
  • Filtering & Sorting: Executes custom predicates for WHERE operators, and sorts variables descending or ascending.
  • Relational JOINs: Combines records from two tables by performing a cartesian product scan and matching rows on specified key equations.
  • Aggregates: Computes inline counts, sums, and averages (e.g. COUNT(*), AVG(years_active)).

🚀 The Local-First Design Philosophy

Building SQL Quest as a zero-dependency, client-side application provides distinct benefits:

  • Zero Latency: Queries run locally in milliseconds, allowing rapid trial-and-error typing.
  • Educational Warnings: Allows custom error hooks that intercept bad syntax and provide specific recommendations.
  • No Server Maintenance: Ensures hosting is completely free forever.
